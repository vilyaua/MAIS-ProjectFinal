# User Story

> Build a REST API with Flask for a task manager with SQLite persistence, user authentication via API keys, and CRUD operations for tasks with priorities and due dates

## Specification: Task Manager REST API with Flask and SQLite

**Complexity:** medium

### Requirements

1. Implement a Flask-based REST API providing endpoints to perform CRUD operations on tasks.
2. Persist tasks in a SQLite database with fields: task id, description, priority, due date, and created/updated timestamps.
3. Implement user authentication via API keys; each API request must include a valid API key to access endpoints.
4. Validate input data for tasks, ensuring priorities follow allowed values (e.g., low, medium, high) and due dates are valid ISO 8601 date strings.
5. Handle error scenarios gracefully, returning appropriate HTTP status codes and messages for authentication failures, invalid data, not found tasks, etc.
6. Support common CRUD operations for tasks: create (POST), read (GET), update (PUT/PATCH), and delete (DELETE).
7. Ensure that the API only allows authenticated users to access and manipulate their own tasks (tasks must be associated with a user).

### Acceptance Criteria

1. Given a valid API key, when a client sends a POST request to /tasks with valid task data, then a new task is created and persisted with status 201.
2. Given an invalid or missing API key, when any endpoint is accessed, then the API returns a 401 Unauthorized response.
3. Given a request with invalid task data (e.g., malformed due date), when the client sends it to the API, then the API returns a 400 Bad Request response.
4. Given a valid API key and task ID, when the client sends a GET request to /tasks/{id}, then the API returns the corresponding task data with status 200.
5. Given a valid API key and task ID, when the client sends a DELETE request to /tasks/{id}, then the task is deleted and the API returns status 204.
6. Given a valid API key and correct task modification data, when the client sends a PUT or PATCH request to /tasks/{id}, then the task is updated accordingly with status 200.
7. Given a valid API key, when a client requests a list of tasks, then the API returns only tasks owned by the API key's user with status 200.
